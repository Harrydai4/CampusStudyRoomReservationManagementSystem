# 问题记录

## 1. 学生端生成签到二维码没有实现

问题：学生端点击“生成签到二维码”后，页面只是展示 token 的前 32 位文本，并不是真正可被扫码识别的二维码；二维码内容也没有明确绑定学生信息。

原因：前端使用 `.fake-qr` 样式伪造二维码视觉效果，没有调用二维码编码逻辑。后端 `/api/checkin/qrcode` 只返回简单 token，没有把学生学号、姓名和预约展示信息一起返回给前端。

解决思路：后端根据当前登录学生、待签到预约、生成时间、学号和姓名生成签到 token，并返回学生和预约信息；前端把 token 编码成真实 QR SVG，同时在二维码下展示学生信息和预约位置，便于管理员核对。

具体解决方法（怎么改代码）：新增 `frontend/src/qr.js`，实现本地 QR SVG 生成方法 `createQrSvg`；修改 `frontend/src/App.vue` 的 `loadQr()`，调用 `/api/checkin/qrcode` 后用 `createQrSvg(data.qrToken)` 生成真实二维码，并新增 `qrInfo` 展示姓名、学号、自习室、座位和有效期；修改 `frontend/src/styles.css`，删除伪二维码样式，新增 `.qr-card`、`.qr-image`、`.qr-meta` 样式；修改 `AppService.qrCode()`，token 内容由 `用户ID:预约ID:时间戳:学号:姓名` 生成，并返回学生与预约信息。

## 2. 管理员端手机签到需要调用摄像头扫码

问题：管理员端签到管理只支持粘贴二维码 token，手机使用时无法直接调用摄像头扫描学生二维码完成签到。

原因：前端签到管理页没有 `getUserMedia` 摄像头调用，也没有二维码识别循环；后端扫码接口虽然存在，但前端没有把摄像头识别结果自动提交给接口。

解决思路：在管理员签到页增加“打开摄像头扫码”入口。移动端浏览器通过 `navigator.mediaDevices.getUserMedia` 打开后置摄像头，并使用原生 `BarcodeDetector` 识别 `qr_code`；识别成功后自动把 token 提交到 `/api/admin/checkin/scan`。同时保留手动粘贴 token 的兜底方式，兼容不支持 `BarcodeDetector` 的浏览器。

具体解决方法（怎么改代码）：修改 `frontend/src/App.vue` 的管理员签到模板，加入摄像头按钮、视频预览、扫描框和提示文案；新增 `scanVideo`、`isScanning`、`scanHint` 等状态，以及 `startCameraScan()`、`scanCameraFrame()`、`stopCameraScan()` 方法；扫描成功后设置 `scanToken` 并调用现有 `scanCheckin()`；切换管理员页面、退出登录和组件卸载时调用 `stopCameraScan()` 释放摄像头；修改 `frontend/src/styles.css`，新增 `.scanner-toolbar`、`.scanner-view`、`.scanner-frame`、`.scanner-hint` 样式；修改 `AppService.scanCheckin()`，解析 token 中的时间戳并校验 60 秒有效期，避免旧二维码被重复使用。

## 3. Maven 测试使用默认 JDK 时版本不匹配

问题：直接运行 `mvnw test` 时，默认系统 Java 是 20，而项目 `pom.xml` 配置 `java.version` 为 21，导致 Maven 编译报“不支持发行版本 21”。

原因：项目要求 Java 21 编译，但系统 PATH 中优先使用的是 `D:\Java` 或 Oracle javapath 下的 Java 20；Maven wrapper 会跟随当前 `JAVA_HOME`/PATH 选择 JDK。

解决思路：不改系统 JDK，改为使用项目内 `.tools` 目录中的 JDK 21 执行 Maven，并把 Maven 用户目录指向项目内 `.m2-home`，保证验证环境可复现。

具体解决方法（怎么改代码）：业务代码不需要修改；执行测试前在 PowerShell 中设置 `$env:JAVA_HOME=(Resolve-Path '.tools\jdk21\jdk-21.0.11+10').Path`，再把 `$env:Path` 前置为 `$env:JAVA_HOME\bin`，同时设置 `$env:MAVEN_USER_HOME=(Resolve-Path '.m2-home').Path`，最后运行 `.\mvnw.cmd test`。

## 4. 项目内 npm 执行构建脚本时出现权限错误

问题：使用项目内 Node/npm 运行 `npm run build` 时，npm 已能输出版本号，但执行脚本阶段出现 `Access is denied.`。

原因：当前沙箱/Windows 执行环境对 npm 脚本间接启动 `vite` 的方式有限制；同一个项目内 Node 直接执行 Vite 入口文件可以正常构建，说明前端代码和依赖本身没有构建错误。

解决思路：保留 `package.json` 的标准构建脚本不改，验证时使用项目内 `node.exe` 直接运行 `node_modules/vite/bin/vite.js`，绕开 npm 脚本启动权限问题。

具体解决方法（怎么改代码）：业务代码不需要修改；验证命令改为在 `frontend` 目录执行 `.tools\node\node-v24.11.1-win-x64\node.exe .\node_modules\vite\bin\vite.js build --outDir ..\src\main\resources\static --emptyOutDir`。

## 5. 管理员端未提交二维码内容时提示“当前无可签到预约”

问题：学生在本机提交预约后，手机管理端点击签到按钮可能提示“当前无可签到预约”，容易误以为预约没有同步到管理端。

原因：管理员扫码接口允许通过 `qrToken`、`userId` 或 `reservationId` 签到；当手机端没有真正扫到二维码、输入框也是空的时候，请求体里没有任何可定位学生或预约的信息。后端继续按空 `userId` 查询待签到预约，查不到后返回“当前无可签到预约”，提示语不准确。

解决思路：在后端扫码签到入口先判断是否拿到了二维码 token 或明确的学生/预约 ID；如果都没有，直接返回“请先扫描或粘贴学生签到二维码”，让管理员知道需要先扫学生端生成的二维码。

具体解决方法（怎么改代码）：修改 `src/main/java/com/scau/campusstudyroomreservationmanagementsystem/service/AppService.java` 的 `scanCheckin()` 方法，在解析 `qrToken` 后增加 `if (userId == null && reservationId == null)` 校验，直接抛出 `BusinessException(400, "请先扫描或粘贴学生签到二维码")`，避免继续执行空条件查询并返回误导性提示。

## 6. 手机管理端点击“扫码签到”没有启动摄像头动作

问题：手机管理端进入签到页后，点击“扫码签到”没有出现摄像头预览或扫码动作。

原因：前端 `startCameraScan()` 先判断浏览器是否支持原生 `BarcodeDetector`，如果不支持就直接返回，导致后面的 `getUserMedia()` 摄像头调用根本没有执行。部分手机浏览器即使可以打开摄像头，也不支持 `BarcodeDetector`，所以表现为“没有动作”。

解决思路：把摄像头调用放到二维码识别能力检测之前。管理员点击“扫码签到”时先打开摄像头和视频预览；如果浏览器支持 `BarcodeDetector`，继续自动识别二维码并提交签到；如果不支持，也明确提示“摄像头已打开，但当前浏览器不支持自动识别二维码”，避免看起来像按钮无效。

具体解决方法（怎么改代码）：修改 `frontend/src/App.vue` 的管理员签到按钮文案，将主按钮改为“扫码签到/停止扫码”，手动提交按钮改为“手动确认”；调整 `startCameraScan()` 顺序，先调用 `navigator.mediaDevices.getUserMedia()`、设置 `isScanning=true`、绑定 `scanVideo.srcObject` 并播放视频，再判断 `BarcodeDetector` 和 `qr_code` 支持情况；支持时创建 `new BarcodeDetector({ formats: ['qr_code'] })` 并执行 `scanCameraFrame()`，不支持时保留摄像头预览并显示明确提示。

## 7. 学生端界面与原型截图风格和交互不一致

问题：学生端页面虽然具备基础功能，但整体 UI 与原型截图差异较大：登录页缺少紫蓝背景和登录卡片层级，首页公告/今日预约/推荐自习室不够接近手机原型，预约页缺少日期卡、快捷时段、座位分布图和分区座位选择，签到、我的、信用积分、学习统计页面也缺少原型中的关键卡片结构。

原因：前端初版更偏向后台式通用表单和列表，使用简单 Element Plus 控件直接堆叠；没有按学生移动端使用流程组织页面，也没有把“预约配置 -> 座位分布 -> 分区座位 -> 确认预约”的交互链路做完整。

解决思路：保留现有后端接口和数据字段，重构学生端模板与样式。页面风格统一为浅蓝背景、白色大圆角卡片、紫蓝主按钮、底部四栏导航；预约页新增日期横向卡片、快捷时段、房间选择、筛选标签、座位分布图和分区座位选择；个人中心、信用积分和学习统计补齐原型里的卡片和统计信息。

具体解决方法（怎么改代码）：修改 `frontend/src/App.vue`，重写学生端登录、首页、预约、签到、我的、信用积分、学习统计、反馈等模板；新增 `dateOptions`、`quickTimeRanges`、`seatGroups`、`seatMapPreview`、`activeStudentTab`、`creditPercent`、`creditLevel`、`studyBars` 等计算属性；新增 `setReservationDate()`、`setQuickTime()`、`formatDate()`、`timeRangeText()`、`changeStatPeriod()` 等交互方法；修改预约成功后跳转到签到页；重写 `frontend/src/styles.css`，实现原型风格的移动端布局、日期胶囊、座位分布图、分区座位按钮、圆形签到按钮、个人中心菜单、积分圆环和学习统计柱状图。

## 8. 手机端字体和组件过大导致真实设备显示效果不好

问题：学生端页面在真实手机浏览器或手机 WebView 中显示时，首页标题、问候卡片、公告卡片、预约页日期卡片、表单控件和底部导航整体尺寸偏大，导致首屏信息密度过低，部分内容被底部导航遮挡或需要频繁滚动，视觉效果不符合手机端使用习惯。

原因：上一版样式为了贴近原型截图，直接放大了移动端字号、圆角、卡片间距和底部导航尺寸；但真实手机浏览器还会占用顶部地址栏和底部系统导航空间，可用视口比原型画布更小，导致组件在实际设备上显得过大。底部导航高度和页面底部留白也没有按真实手机视口重新压缩。

解决思路：保留原型的浅蓝背景、白色卡片、紫蓝主色和底部四栏导航风格，但在小屏断点下整体降低字号和控件高度，减少卡片 padding、圆角、间距和底部导航高度，让首页、预约页、签到页、我的、信用积分、学习统计等页面在手机端更紧凑，并确保底部导航不会遮挡主要内容。

具体解决方法（怎么改代码）：修改 `frontend/src/styles.css` 中 `@media (max-width:520px)` 的移动端样式，压缩 `.topbar` 高度与标题字号，降低 `.content` 上下内边距并调整底部留白；缩小 `.hero-card`、`.announcement`、`.today-card`、`.room-card`、`.date-pill`、`.reserve-config`、`.quick-times`、`.filter-row`、`.seat-map-preview`、`.seat-choice`、`.timer`、`.round-action`、`.profile-card`、`.menu-list`、`.credit-ring`、`.timeline-item`、`.stats-tabs` 和 `.bar-chart-lite` 等组件的字号、高度、间距和圆角；同时把 `.bottom-nav` 高度从偏大的样式压缩为 66px，图标和文字同步缩小，避免真实手机端出现导航栏过厚和内容遮挡。

## 9. 座位预约页存在冗余时间设置、臆造座位分区和交互缺失

问题：座位预约页的“选择时段”和“预约配置”里同时出现时间设置，形成冗余；时间选择步长仍是 30 分钟，不能按 10 分钟选择；结束时间没有在前端明确限制必须晚于开始时间；页面里出现“快捷时段”“精品区”“静音区”等详细设计文档没有描述的分类；点击座位后没有先展示座位配置；我的预约列表没有直观显示每条预约的中文状态；首页通知按钮太宽；点击公告只弹出简短提示，没有展示公告内容；座位分布图也不应该由前端伪造，而应该使用管理端上传的图片。

原因：上一版为了快速贴近原型视觉，额外添加了快捷时段、筛选标签和分区座位分组，但这些交互没有来自详细设计文档；前后端时间槽仍按 30 分钟处理；公告点击只复用了 toast 提示；预约记录卡片只展示房间、时间和座位，没有状态字段；座位图使用前端模拟布局，没有读取自习室的 `layout_image_url` 字段。

解决思路：回到详细设计描述的功能边界，保留必要的日期、开始时间、结束时间、自习室、座位选择和确认预约流程。时间选择统一放在“选择时段”区域，步长改为 10 分钟，并在前端和后端同时校验结束时间晚于开始时间；删除快捷时段和人为分区，只展示完整座位列表；座位点击后弹出配置说明，再由学生确认选择；公告和座位详情使用弹窗展示；座位分布图读取管理端维护的图片地址；我的预约卡片补充中文状态。

具体解决方法（怎么改代码）：修改 `frontend/src/App.vue`，将两个 `el-time-select` 的 `step` 改为 `00:10`，新增 `endSelectStart`、`timeToMinutes()`、`addMinutes()`、`ensureEndAfterStart()`、`handleStartTimeChange()` 和 `handleEndTimeChange()`，动态限制结束时间必须晚于开始时间；删除 `quickTimeRanges`、`setQuickTime()`、`seatFilters`、`seatGroups`、`seatMapPreview` 以及对应模板；座位列表改为直接遍历 `seats`，点击时调用 `openSeatDetail()` 打开座位配置弹窗，确认后调用 `confirmSeatSelection()` 选中座位；新增公告详情弹窗 `announcementDetailOpen` 和 `activeAnnouncement`，`readAnnouncement()` 改为标记已读后展示公告标题、内容和时间；`ReservationCard` 新增 `statusText` 入参并渲染中文状态标签；自习室表单新增 `layoutImageUrl` 字段，预约页使用 `currentRoom.layout_image_url` 渲染 `.seat-layout-image`，没有图片时显示“管理员尚未上传”；修改 `frontend/src/styles.css`，缩小首页铃铛按钮、增加座位配置弹窗、预约状态和上传座位图样式；修改 `AppService.saveRoom()` 保存 `layout_image_url`，并把 `slots()`、预约槽 `slot_end` 和 `validateTimeRange()` 从 30 分钟对齐调整为 10 分钟对齐。

## 10. 学习统计周报月报标题不准确且柱状图底部不对齐

问题：学习统计页面切换到周报或月报时，图表标题仍显示“今日各时段学习时长（小时）”，语义不正确；柱状图里数值、柱体和底部标签共同参与同一个网格排版，导致蓝色柱子的底部没有落在同一条基线上，视觉上参差不齐。

原因：前端 `studyBars` 使用固定的“小时段”演示数据，不区分日报、周报和月报；图表模板直接把数值、柱体和横轴标签依次放在 `.bar-col` 中，柱体高度变化会影响同列内容的排布。后端 `/statistics/my-study-duration` 也没有真正按 `period` 区分日报、周报和月报数据。

解决思路：日报显示“今日各时段”，周报显示“本周每日”，月报显示“本月每日”；前端柱状图每一列固定为“数值区 + 等高柱体轨道 + 标签区”，让柱子始终从同一条底线向上增长；后端按 `period` 返回对应粒度的数据，避免周报月报继续使用今日小时段的语义。

具体解决方法（怎么改代码）：修改 `frontend/src/App.vue`，新增 `studyChartTitle` 根据 `statPeriod` 动态显示标题；重写 `studyBars`，优先使用接口返回的 `series`，日报标签格式化为“xx时”，周报/月报标签格式化为日期；新增 `maxStudyBarValue` 和 `barHeight()`，按最大值计算柱高比例；模板中为每个柱子新增 `.bar-track` 等高轨道，柱体放在轨道内从底部对齐。修改 `frontend/src/styles.css`，将 `.bar-chart-lite` 改为横向自适应列布局，将 `.bar-col` 固定为三行，把 `.bar-track` 设置为固定高度并用 `align-items:flex-end` 保证柱子底部对齐。修改 `AppService.myStudyDuration()`，日报按当天 `hour(sign_in_time)` 聚合，周报按最近 7 天 `reserve_date` 聚合，月报按最近 30 天 `reserve_date` 聚合。
